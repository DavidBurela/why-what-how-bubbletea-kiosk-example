# BDD Enhancements — Documentation, Reporting, and Verification Linking

Enrich the BDD documentation from a thin initial draft into a proper showcase of how BDD bridges business scenarios to executable verification. Add a test report pipeline (HTML for rich viewing + generated markdown for GitHub rendering). Link verification status from business scenarios and business context.

## Decisions

| # | Decision | Rationale |
|---|----------|-----------|
| D1 | `docs.why-what-how.bdd.md` is portable — no project-specific references | This file is copied across projects using the Why-What-How framework. .NET execution details belong in `bdd-conventions.md` under standards. |
| D2 | Both HTML and markdown reports committed | HTML provides rich interactive viewing when the repo is cloned. Markdown renders on GitHub without cloning. Both are regenerated by the script on demand. |
| D3 | Verification lives under `docs/business/verification/` | Verification status is business-layer content — black-box proof that scenarios hold. Not solution-layer. |
| D4 | Business → BDD status linking is allowed | Per `docs.conventions.md` linking direction rules: Business → BDD tests is same-layer (both black-box). Verification status follows the same principle. |
| D5 | No paid tools — built-in `dotnet test` loggers | TRX + HTML loggers are included with `dotnet test`. No Reqnroll.Plus.LivingDoc or other paid dependencies. The script parses TRX XML to generate the markdown summary. |
| D6 | Report shows timestamp, not live status | The markdown summary reflects the last manual run. A note makes this explicit. CI automation can be added later. |
